Workshop Hands-On Tasks

1 · Observe CRITICAL risk band with default scenario

Run Supplier Delay + Port Congestion with 12 batches. Watch the risk score climb to 85–90. The root cause: shipment delay (96h) exceeds available inventory coverage (0.87 days of supply).

Scoring breakdown: Inventory 35 + Shipment 30 + Supplier 15 + Customer 20 + External 20 − Mitigation 10 = Score 90

2 · Show the recovery transition

Switch to the Recovery scenario and run with 20 batches. After batch 10, delay drops to 12h and inventory restores. Watch the risk band transition: CRITICAL → HIGH → MEDIUM. The alternate supplier (SUP-221, reliability 91) contributes −10 mitigation credit.

3 · Examine per-factor scoring

After running a scenario, open the Analytics tab. The factor breakdown chart shows exactly which driver (inventory, shipment, external) is contributing most to the score. The trend chart shows how risk evolved per event batch.

4 · Generate an AI recommendation

Open the AI Prompts tab after running a scenario. Click Executive Summary, Supplier Email, or Procurement Escalation. These templates use the live risk and recommendation data. In production, they would be sent to watsonx.ai for LLM generation.

5 · Inspect the Kafka topic model

Open the Topics & Schemas tab. It shows all 10 Kafka topics (7 input + 3 output), their keys, and the JSON schema for each. These schemas are deployable to Confluent Schema Registry with one command: python -m scrc.register_schemas

Risk Scoring Logic (Python → JavaScript)

The scoring formula is fully transparent. The same logic runs in src/scrc/risk_logic.py, Flink SQL, and this UI's JavaScript engine.

score = inventory_risk       (0–35)
      + shipment_delay_risk  (0–30)
      + supplier_risk        (0–20)
      + customer_impact_risk (0–20)
      + external_event_risk  (0–20)
      − mitigation_credit    (0–10)
      → capped to [0, 100]

RISK BAND
  score ≥ 85  →  CRITICAL
  score ≥ 70  →  HIGH
  score ≥ 40  →  MEDIUM
  score  < 40  →  LOW

INVENTORY RISK
  days_of_supply ≤ 0                       → 35
  days_of_supply < max(delay_days, 1)      → 35
  days_of_supply < 3                       → 25
  days_of_supply < 7                       → 15
  else                                     →  0

SHIPMENT DELAY RISK
  min(30, floor(delay_hours / 3))

SUPPLIER RISK
  reliability < 70  → 20
  reliability < 80  → 15
  reliability < 90  →  8

CUSTOMER IMPACT
  priority = STRATEGIC           → 20
  priority = HIGH                → 15
  revenue_at_risk ≥ $250,000     → 10

EXTERNAL EVENT RISK
  min(20, severity × 4 + floor(expected_delay_hours / 12))

MITIGATION CREDIT
  alternate supplier available   → −10
  component criticality LOW/MED  → −5
Architecture

Building Block Architecture

End-to-end logical architecture from enterprise source systems through Confluent Cloud to downstream AI, analytics, and operational sinks.

Design Principles

Event-First

Every change — supplier update, shipment delay, inventory movement, customer order — is treated as an event. This enables real-time correlation across systems that were never designed to talk to each other.

Transparent Scoring

Risk scores are calculated from explainable factors, not a black-box ML model. Each factor is visible in the risk card. This makes the score easy to explain to a supply chain leader or procurement manager.

Governed

JSON schemas and data contracts in Confluent Schema Registry enforce event shape, required fields, and valid enum values before data is consumed by the risk engine or any downstream system.

Composable

Source systems are replaceable. The synthetic producers can be swapped for Confluent managed connectors (JDBC for Db2, MQ bridge for IBM MQ, HTTP source for supplier APIs) without changing the scoring layer.

Contract Rules by Topic

            

              

                
supplier_profiles

                
supplier_id required · reliability_score 0–100 · risk_tier must be LOW | MEDIUM | HIGH

              

              

                
shipments

                
shipment_id, po_id, component_id, supplier_id, eta, delay_hours (≥0), route_id required · status must be BOOKED | PICKED_UP | IN_TRANSIT | AT_PORT | CUSTOMS_HOLD | DELAYED | DELIVERED

              

              

                
external_risk_events

                
event_type must be WEATHER | PORT | TARIFF | GEOPOLITICAL | SUPPLIER_CAPACITY | TRANSPORT | FUEL_COST · severity must be integer 1–5

              

              

                
customer_orders

                
priority must be LOW | MEDIUM | HIGH | STRATEGIC · revenue_at_risk must be a number ≥ 0

              

            

          

          

            
Schema Evolution Policy

            

              
Safe changes
Adding optional fields is always backward-compatible.

            

            

              
Requires versioning
Renaming or removing required fields. Changing enum values. Changing field semantics.

            

            

              
Never do
Changing a required field's type without a transition period. Removing a field with existing downstream consumers.

            

            

              
Production metadata to add
data owner · domain owner · freshness SLO · PII tags · downstream consumers · escalation contact
